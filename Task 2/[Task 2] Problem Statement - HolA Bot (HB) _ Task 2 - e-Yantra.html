Problem Statement

Task 1, in short:
Implement a simple controller to make the simulated robot (simplified) go to a series of desired pose given simulator transform as localisation.

Task 2 in short (as mentioned in welcome page):
Repeat Task 1! But…

  • with more realistic localisation:
  • with more realistic model of controlling the holonomic drive.

The gif below is a reminder of what one should be able to do at the end of Task 1.

Approach

So how shall we tackle Task 2?

We have made some changes to the model (that we had been working with in task 1) and the world to make interfacing with it more realistic.

Specifically,

  • we have modified the urdf and used a slightly customised plugin to control wheel velocities (v1, v2, v3) like a real robot instead of directly controlling velocities of the robot chassis (v_x, v_y, w).
  • And we have also added an Aruco marker and a camera to do localisation like we would in stage 2.

All the above-mentioned setup along with the evaluator for (auo-evaluation) should be downloaded from the link below:

The RUN :

(common step : source ~/catkin_ws/devel/setup.bash)

  • roslaunch your_package_name gazebo.launch

  • rosrun your_package_name feedback.py

  • rosrun your_package_name task2exe your_package_name

  • At the end of RUN, task2exe node generates a task2_output.txt in the same folder, which will be required for submission.

Now, coming to the approach to complete the task, let’s look at the checklist above and work on it one by one.

Part A: Localisation with OpenCV and Aruco Markers 315

  • Write your code in the feedback.py file.
  • using Overhead camera and Aruco Marker instead of simulator transforms

Part B: Inverse Kinematics

  • Write your code in the controller.py file.
  • More than half of the work in controller.py was already done in Task 1, the difference is:
    • pose is given in pixels and radians (instead of meters and radians) by subscribing to the *detected_aruco topic (NOTE: feedback.py publishes to this topic)
    • one Matrix Multiplication needs to be implemented for inverse kinematics and
      • i.e. find three omni-wheel velocities (v1, v2, v3)
        given velocity of the chassis (Vx, Vy, W)
    • publish to the three wheel velocities in the three publishers already declared in the boiler plate code given.
    • Additionally, some of you had a question about the bot continues to move even after terminating the controller, etc.
      Question: Can you do something about that given the “signal_handler” and “cleanup” functions given in the boiler plate?

That is all! Complete the two python files: feedback.py & controller.py and
if you are able to repeat task 1

  • but by only using Aruco markers for feedback(on topic /detected_aruco) instead of /odom (gazebo transforms)
  • and controlling wheel velocities(on topics /front_wheel_force, /right_wheel_force, /left_wheel_force) instead of /cmd_vel (chassis velocities)

then Congratulations on completing Task 2!

Note:
Exploring the downloaded package and reading the boilerplate carefully should give you enough boost to start working on the task (the two py files).

You’ll need to explore OpenCV to figure out how to detect Aruco marker, and you’ll need to explore some paper or other resources to learn about how to get wheel velocities (v1, v2, v3) from chassis velocities (v_x, v_y, w).
